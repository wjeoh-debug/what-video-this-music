# Vibe Plan — 기획 → Feature Progress 변환 스킬

## 역할

`_docs/PRD.md`의 기획 원문을 분석하여 `_docs/TASKS.md`를 생성하거나 갱신한다.
비개발자가 이해할 수 있는 언어로 기능을 분해하고, 구현 순서에 맞는 체크박스를 만든다.

---

## 실행 순서

### Step 1. _docs/PRD.md 읽기

`_docs/PRD.md` 파일을 읽고 다음을 파악한다:
- 서비스의 핵심 목적 (1문장으로 요약)
- 주요 사용자가 누구인지
- 구현해야 할 기능 목록 (명시적/암묵적 모두 파악)
- 우선순위 또는 구현 순서 힌트

### Step 2. 기능 분해

각 기능을 구현 가능한 서브태스크로 분해한다:

**규칙:**
- 기능당 최소 3개 / 최대 7개 서브태스크
- 구현 순서대로 나열 (화면 구성 → 데이터 연결 → 동작 구현 → 디자인 → 검수)
- 비개발자가 이해할 수 있는 단어 사용
  - ✅ "목록 화면 만들기", "버튼 클릭 동작 구현", "디자인 입히기"
  - ❌ "API 엔드포인트 구현", "컴포넌트 리팩토링"
- 각 기능의 Status는 첫 번째 기능만 `🔄 진행 중`, 나머지는 `⏳ 예정`

### Step 3. _docs/TASKS.md 생성

아래 형식으로 파일을 생성한다:

```markdown
# [프로젝트명] Feature Progress

## 📌 프로젝트 개요
- **서비스 목적**: [1문장 요약]
- **주요 사용자**: [대상]
- **시작일**: [YYYY-MM-DD]

---

## 📊 진행 현황
> 마지막 업데이트: [YYYY-MM-DD]

| 구분 | 수 |
|------|-----|
| 전체 기능 | N |
| ✅ 완료 | 0 |
| 🔄 진행 중 | 1 |
| ⏳ 예정 | N-1 |

---

## 🚀 기능 목록

### [F1] 기능명
> **Status**: 🔄 진행 중
> **목표**: 사용자 관점에서 이 기능이 무엇을 해주는지 1문장

- [ ] 서브태스크 1
- [ ] 서브태스크 2
- [ ] 서브태스크 3

---

### [F2] 기능명
> **Status**: ⏳ 예정
> **목표**: ...

- [ ] 서브태스크 1
- [ ] 서브태스크 2

---

## 💡 백로그 (추가 아이디어)
> 구현할지 미정인 아이디어. 확정되면 위 기능 목록으로 이동.

(현재 없음)

---

## 🐛 알려진 이슈
(현재 없음)

---

## 🎯 완료 기준
> 위 기능 목록의 모든 체크박스가 `[x]`가 되면 프로젝트 완료
```

### Step 4. 확인 및 시작

생성 후 아래 형식으로 요약 보고:

```
✅ _docs/TASKS.md 생성 완료!

📋 기능 목록 (총 N개):
  🔄 F1. [기능명] — N개 태스크
  ⏳ F2. [기능명] — N개 태스크
  ⏳ F3. [기능명] — N개 태스크

F1 "[기능명]"부터 시작할까요?
첫 번째 태스크: "[서브태스크 1]"
```

---

## 기존 _docs/TASKS.md가 있을 때

이미 진행 중인 프로젝트에서 `/vibe-plan`을 실행하면:
- 완료된 기능(`✅`)은 건드리지 않는다
- 새 기능 요청이 있으면 기능 목록 끝에 추가한다
- 백로그 항목을 기능 목록으로 승격시킬 수 있다
- 진행 현황 표를 업데이트한다
